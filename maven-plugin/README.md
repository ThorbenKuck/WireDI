# WireDI - Maven Plugin

A challenge when using WireDI is to set up the annotation processors correctly.
It is important to configure the annotation processors correctly, because otherwise code will not be generated correctly.
Additionally, you need to manage all the plugins for the annotation processors correctly.

This plugin aims to solve this challenge by providing a simple way to configure the annotation processors.

## Usage

To use this plugin, add it to the build section of your project.
It looks like this:

```xml
<build>
    <plugins>
        <plugin>
            <groupId>com.wiredi</groupId>
            <artifactId>maven-plugin</artifactId>
            <version>1.0.0</version>
            <!-- Important! -->
            <!-- This enables the plugin to configure the project for you -->
            <extensions>true</extensions>
        </plugin>
    </plugins>
</build>
```

The version can be simply the same as the WireDI version.
With this plugin present, you don't need to configure the annotation processors manually.
Running `mvn help:effective-pom` will show you the configured annotation processor.

```xml
<dependencies>
    ...

    <dependency>
        <groupId>com.wiredi</groupId>
        <artifactId>processors</artifactId>
        <version>1.0.0</version>
        <scope>provided</scope>
    </dependency>
</dependencies>
```

Additionally, the plugin will add the `wiredi-processor` to the annotation processor path by configuring the compiler plugin.
You can see this in your plugin section:

```xml
<plugins>
    <!-- Other plugins -->
    ...
    
    <plugin>
        <artifactId>maven-compiler-plugin</artifactId>
        <version>3.13.0</version>
        <executions>
            <execution>
                <id>default-compile</id>
                <phase>compile</phase>
                <goals>
                    <goal>compile</goal>
                </goals>
            </execution>
            <execution>
                <id>default-testCompile</id>
                <phase>test-compile</phase>
                <goals>
                    <goal>testCompile</goal>
                </goals>
            </execution>
        </executions>
        <configuration>
            <annotationProcessorPaths>
                <dependency>
                    <groupId>com.wiredi</groupId>
                    <artifactId>processors</artifactId>
                    <version>1.0.0</version>
                </dependency>
            </annotationProcessorPaths>
        </configuration>
    </plugin>
</plugins>
```

If present, the plugin will also configure the Kotlin KAPT plugin and skip configuring the Java compiler plugin.
If you where to configure both plugins, you'd have errors with generated code.
Classes generated by the Java compiler plugin would not be touched by the kotlin compiler plugin.

So, if you use Kotlin, you don't need to configure the annotation processors manually as well.

### Plugins

The annotation processors support plugins that can enhance how the generated code is processed.
You can configure these plugins by adding them to the `compilerPlugins` section of the plugin configuration.

As an example, let's assume you want to add bother the web and the jpa plugins.
You'd do this like this:

```xml
<build>
    <plugins>
        <plugin>
            <groupId>com.wiredi</groupId>
            <artifactId>maven-plugin</artifactId>
            <version>${wiredi.version}</version>
            <!-- Important! -->
            <!-- This enables the plugin to configure the project for you -->
            <extensions>true</extensions>
            <!-- Configure the plugins -->
            <configuration>
                <!-- Add the plugins here -->
                <compilerPlugins>
                    <plugin>
                        <groupId>com.wiredi.web</groupId>
                        <artifactId>plugin</artifactId>
                        <version>1.0.0</version> <!-- Optional! See further down what this entails -->
                    </plugin>
                    <plugin>
                        <groupId>com.wiredi.jpa</groupId>
                        <artifactId>plugin</artifactId>
                        <version>1.0.0</version> <!-- Optional! See further down what this entails -->
                    </plugin>
                </compilerPlugins>
            </configuration>
        </plugin>
    </plugins>
</build>
```

Looking at the effective POM again, you can see that the annotation processor path now contains the web and jpa plugins as well:

```xml
<build>
    <plugins>
        <plugin>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.13.0</version>
            <executions>
                <execution>
                    <id>default-compile</id>
                    <phase>compile</phase>
                    <goals>
                        <goal>compile</goal>
                    </goals>
                </execution>
                <execution>
                    <id>default-testCompile</id>
                    <phase>test-compile</phase>
                    <goals>
                        <goal>testCompile</goal>
                    </goals>
                </execution>
            </executions>
            <configuration>
                <annotationProcessorPaths>
                    <dependency>
                        <groupId>com.wiredi</groupId>
                        <artifactId>processors</artifactId>
                        <version>1.0.0</version>
                    </dependency>
                    <dependency>
                        <groupId>com.wiredi.web</groupId>
                        <artifactId>plugin</artifactId>
                        <version>1.0.0</version>
                    </dependency>
                    <dependency>
                        <groupId>com.wiredi.jpa</groupId>
                        <artifactId>plugin</artifactId>
                        <version>1.0.0</version>
                    </dependency>
                </annotationProcessorPaths>
            </configuration>
        </plugin>
    </plugins>
</build>
```

### Versioning

The version of each compiler plugin can be omitted.
If omitted, the maven-plugin will attempt to resolve the latest version of the plugin.

> [!NOTE]
> When resolving the latest version, the maven-plugin will only consider versions that are not SNAPSHOT, alpha, beta or RC.
